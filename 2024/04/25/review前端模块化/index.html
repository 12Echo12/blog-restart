关于 JS 的执行环境: node.js VS 浏览器（插播一段）

它们两个都使用 JS 作为编程语言，主要区别如下：

  • 所做的操作不同：

    • 在浏览器中我们大多要与 DOM 进行一些交互。

    • 而在 node 中我们更多要做的是读写文件、访问数据库等操作。

  • 拥有的对象不同：

    • 出于操作DOM的初心，浏览器给我们提供了document、window 等对象
    • 而 node 并没有上述两个对象，取而代之的是 global 。但 node 为我们内置了许多优秀的模块，如文件系统…

  • 对模块化的支持不同：

    • 浏览器前比较广泛地支持ES Modules。但因为ES Modules是 ES6 新特性，所以有些版本较低的浏览器不支持，所以也需要我们用 babel 对其进行转译，向后兼容”翻译”为浏览器可以识别的语法。
    • node (从v12开始) 同时支持 CommonJS 和 ES Modules

回到问题，为什么大多数浏览器已经支持了 ES Modules ，也就是说可以在 js 文件里使用 import且在较新版本的浏览器上应该是可以正常执行的。但是为啥就是不执行。真相只有一个 —> 你没按规则使用呗(如果有人读到这里，不好意思，比较喜欢用第二人称骂自己，没有一点骂你们的意思啊哈哈)

那来看看 import 都有哪些要遵守的点：

  • < script type=”module” > 告诉浏览器用处理模块的模式执行该文件

  • 模块只通过 HTTP(s) 工作，而非本地。即如果用 file:// 协议在本地打开一个网页，import/export 是不起作用的。（昨天问题的根源）

    • 解决方法，把代码在 web 服务器上跑起来，可以 npm run start（如果配置了有的话），或者用编辑器的扩展插件（eg：vscode上的 Live Server Extension ）

  • 不允许裸模块 即 import 必须给出相对或绝对的 URL 路径。

    • import {sayHello} from ‘sayHello’ 是不行的。import {sayHello} from ‘./sayHello’是行的嘿嘿
    • 好，我又有疑问，那我平时直接 import React from ‘react’ 怎么说？其实还是打包工具帮我们转译了才得以正常运行哈哈

ES Modules VS CommonJS

不同点：

  • 使用语法不一样：
    • ES Modules 使用 export 和 import 导入导出
    • CommonJS 使用 module.exports 和 require 导入导出
  • 导出的值不一样
    • ES Modules 导出的是值的引用
    • CommonJS 导出的的是值的拷贝，也就是说一旦导出，那模块内部再怎么变也不会影响已经导出的内容。
  • 导入方式不一样
    • ES Modules 是静态导入，即在编译时就会确定模块之间的依赖关系
    • CommonJS 是动态导入，即是在运行时确定模块之间的依赖关系
