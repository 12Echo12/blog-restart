前端要管理的状态可分为两类：

(1) 用户交互的中间状态，比如 弹窗的关闭与否isOpen …

  • 状态以同步的形式更新
  • 往往只在所在组件使用，比较独立 ，经常用useState()。当需要在组件间传递时，可以用 Context API或Redux。
  • 完全由前端控制

(2) 后端返回的数据

  • 数据以异步的形式更新
  • 常常会在组件间通用，所以常常用 Redux 来管理。

所以用户交互的中间状态比较方便管理，而从后端返回的数据直接用 redux 来管理会有点问题，下面是一段常见的从后端获取数据的代码：

(1) 需要重复请求并处理请求中间状态。如果我们在多个组件都需要知道当前请求的状态，那么 const [isLoading, setLoading] = useState(false); 这样的代码就要写 n 遍。

(2) 数据的更新与缓存。不好控制重新请求更新缓存。

而使用 react-query 可以优雅地解决上述问题，（react-query 其实现在叫 tanstack-query ，后面它又兼任了其他框架，所以改名为 tanstack-query 。）

关于 react-query

react-query 的核心是基于 queryKey 管理查询缓存。如果请求的发起依赖于某一变量，那么可以将其包含在 queryKey数组中。queryKey标记着唯一的query，如果多个组件请求同一个 query ，会合并只请求一次。

(1) useQuery() 执行数据查询操作

接收三个参数 queryKey、queryFn 、options对象

  • queryKey 上文讲过

  • queryFn 实际要进行的查询操作

  • options对象，对该 query请求进行配置，常用配置有：

    • staleTime 重新获取数据的时间间隔 默认0
    • cacheTime数据缓存的时间，即再次拉请求更新缓存的时间间隔
    • retry 失败的重试次数
    • enabled 指定是否进行查询，默认为 true ，如果赋值为false 的话，则不会进行查询操作

useQuery() 返回一个查询状态对象，常用的属性：

  • isLoading 为 true 的话就是在查询中
  • isSuccess 查询是否成功
  • isErroe 查询是否失败
  • data 查询得到的数据

(2) useQueryClient() 返回 queryClient实例 常与缓存进行交互

常用的 queryClient.getQueryState(queryKey).data 用来查询指定 query 的缓存数据。

(3) useMutation() 执行数据变更(增删改)

接收两个参数 mutationFn 、options对象

  • mutationFn: 一个异步函数，进行实际的数据变更操作

  • options：配置对象，常用配置有：

    • onMutate: 一个回调函数，在 mutation 开始之前同步执行。
    • onSuccess: 一个回调函数，在 mutation 成功后执行。
    • onSettled: 一个回调函数，在 mutation 结束后执行，无论是成功还是失败。该函数接收 mutation 的结果或错误对象作为参数，可以在该函数中进行收尾工作，如清除加载状态、重置表单等
